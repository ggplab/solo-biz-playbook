# Publication Safety Rules

> 공개 플레이북의 목적은 "실제로 해본 운영 노하우"를 전달하는 것입니다. 모든 것을 지우면 신뢰도와 브랜딩이 사라지고, 너무 자세히 쓰면 고객·계약·개인 인프라가 노출됩니다. 이 문서는 그 경계를 정합니다.

## 공개 판단 원칙

| 등급 | 처리 | 기준 |
|---|---|---|
| **Keep** | 그대로 공개 | 이미 공개 브랜딩에 쓰는 이름, 공개 제품명, 범용 도구명, 구조적 수치 |
| **Generalize** | 일반명사로 변경 | 실제 고객/계약/프로젝트는 아니지만 조합하면 정체가 추정되는 업무명 |
| **Mask** | 역할명·범주명으로 대체 | 고객명, 계약 상대, 비공개 프로젝트명, 내부 채널명, 로컬 인프라 별칭 |
| **Remove** | 삭제 | API 키, 토큰, ID, 실제 경로, 세션 ID, 개인 이메일, 비공개 저장소명 |

목표는 "누가 봐도 배울 수 있지만, 특정 거래·고객·내부 시스템은 재구성할 수 없는 상태"입니다.

## 공개해도 좋은 것

브랜딩과 신뢰에 도움이 되는 정보는 남깁니다.

- 공개 브랜드명: `GGPLab`, 공개 홈페이지, 공개 SNS 계정
- 공개 저작물·강의·오픈소스 저장소처럼 이미 홍보 중인 자산
- 범용 도구명: GitHub, Claude, Codex, Python, Node.js, Google Calendar 등
- 구조적 수치: 토큰 수, 시간, 세션 수, 비율, 비용 환산값
- 공개 가능한 업무 범주: 교육, 콘텐츠, 자동화, 리서치, 멘토링, 전환 페이지
- 방법론과 판단 기준: 체크리스트, 운영 원칙, 계산식, 템플릿 구조

단, 공개 브랜드와 비공개 거래가 같은 문장에 붙으면 일반화합니다.

## 일반화할 것

| 원문 성격 | 공개 표현 예시 |
|---|---|
| 출판사·고객사·계약 상대 | `출판사 A`, `고객사 B`, `계약 상대` |
| 특정 강의 플랫폼·멘토링 플랫폼 | `교육 채널 A`, `멘토링 채널 B` |
| 내부 프로젝트명 | `전환 페이지 프로젝트`, `교육용 웹앱`, `자동화 콘텐츠 라인` |
| 특정 콘텐츠 채널명 | `영상 채널`, `소셜 채널`, `콘텐츠 채널` |
| 개인 서버·장비 이름 | `개인 인프라`, `로컬 서버`, `작업 장비` |
| 특정 인증 문제 | `이메일/문서/캘린더/알림 도구 인증` |
| 계약·금액·조건 | 구조만 설명하고 상대와 조건은 제거 |

## 반드시 제거할 것

- API 키, OAuth 토큰, webhook URL, session ID
- 실제 이메일 주소, 전화번호, 주소, 캘린더 ID, 데이터베이스 ID
- 비공개 GitHub repo, Notion DB, Supabase project ref, Drive folder ID
- 로컬 절대경로: `/Users/name/...`, `/home/name/...`
- 고객명, 미공개 파트너명, 미공개 출판/강의/계약명
- 원본 로그에 포함된 개인명, 파일 경로, 채팅 ID

## 예시

좋은 공개 문장:

> 교육용 웹앱 구축에 218M 토큰이 들어갔고, 인증·권한·로그인 UI가 주요 작업이었다.

피해야 할 문장:

> 고객사 X의 `project-y` Supabase ref `abcd1234`에서 Auth Edge Function을 수정했다.

좋은 공개 문장:

> 장기 콘텐츠 계약 검토일에 단일 최대 토큰 사용량이 발생했다.

피해야 할 문장:

> 출판사명, 도서명, 계약 조항, 회신 메일 제목을 함께 공개한다.

## 푸시 전 점검 절차

1. 새 문서·차트·이미지에 비공개 고객/계약/프로젝트 단서가 있는지 읽습니다.
2. 구조적 수치와 운영 인사이트는 남기고, 상대·장소·계정·경로만 일반화합니다.
3. 아래 스크립트를 실행합니다.

```bash
python3 scripts/public_safety_check.py
```

4. `BLOCK`이 있으면 푸시하지 않습니다.
5. `REVIEW`는 의도적으로 공개할 브랜딩인지 확인합니다.
6. 의도적으로 남기는 항목은 이 문서의 "공개해도 좋은 것" 기준에 맞는지 확인합니다.

## 자동 점검 hook 설치

로컬 저장소에서 한 번만 실행합니다.

```bash
bash scripts/install_public_safety_hook.sh
```

이후 `git push` 때마다 `python3 scripts/public_safety_check.py --all`이 자동 실행됩니다. `BLOCK`은 푸시를 막고, `REVIEW`는 경고만 출력합니다.

자동 hook은 로컬 `.git/hooks/pre-push`에 설치되므로 GitHub에는 올라가지 않습니다. 대신 설치 스크립트만 버전관리합니다.

## 판단 기준

아래 질문 중 하나라도 "예"라면 일반화하거나 제거합니다.

- 이 문장으로 특정 고객·계약·거래 상대를 추정할 수 있는가?
- 여러 문서의 단서를 조합하면 비공개 프로젝트가 드러나는가?
- 이 값이 복사되면 외부 시스템에 접근하거나 식별할 수 있는가?
- 공개 브랜드 홍보보다 사생활·계약 노출 위험이 더 큰가?

반대로 아래 조건을 만족하면 남겨도 됩니다.

- 이미 공개 프로필이나 홈페이지에서 쓰는 브랜딩이다.
- 독자가 따라 하는 데 필요한 구조적 정보다.
- 실제 상대나 내부 식별자를 재구성할 수 없다.
